# 第八步：结构化输出

## 1. 本节目标

这一节要把 Agent 从：

- 只会返回自然语言

升级成：

- 能返回固定 JSON 结构

我们本节新增的具体能力是：

- 输入 `/plan 主题`
- 返回一个 3 天学习计划的 JSON

## 2. 是什么

结构化输出，指的是让模型不要只返回“看起来像答案”的文本，而是返回：

- 程序能直接处理的数据结构
- 例如 JSON、列表、对象字段

在真实 Agent 系统里，结构化输出非常关键，因为后续很多能力都建立在它之上：

- Tool Calling
- Workflow 编排
- 前端展示
- 自动评测

## 3. 为什么

自然语言对人类友好，但对程序不稳定。

比如模型可能会这样回答学习计划：

- 一次写成段落
- 一次写成列表
- 一次写成表格

这对人能看懂，但对程序来说很难稳定处理。

所以我们要让模型输出固定结构，比如：

```json
{
  "topic": "RAG",
  "target_level": "能够理解并实现基础 RAG",
  "days": [],
  "tips": []
}
```

这样后面你才能：

- 直接保存
- 直接渲染
- 直接校验字段

## 4. 怎么做

### 第一步：定义你想要的结果结构

这一节我们先定一个很简单的结构：

- `topic`
- `target_level`
- `days`
- `tips`

其中 `days` 里再放：

- 第几天
- 当天目标
- 任务列表
- 当日产出

### 第二步：在 Prompt 里把返回格式写清楚

你不能只说“帮我做个计划”。

而要明确告诉模型：

- 必须返回 JSON
- 必须有哪些字段
- 不要输出额外解释
- 不要输出 Markdown 代码块

### 第三步：程序端做 JSON 解析

模型说自己返回了 JSON，不等于它一定真的合法。

所以程序必须再做一次：

- 去掉可能的代码块包裹
- `json.loads(...)` 解析

这一步很重要，因为它是：

**模型输出 -> 程序对象**

的转换点。

### 第四步：加一个清晰的命令入口

我们现在用：

```text
/plan RAG
```

来触发结构化输出。

这样你能清楚区分：

- 普通聊天
- 结构化任务

## 5. 关键代码

### 代码 1：提取公共消息构造逻辑

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def _build_messages(self, user_input: str) -> list[dict[str, str]]:
    messages = [{"role": "system", "content": self.system_prompt}, *self.history]
    messages.append({"role": "user", "content": user_input})
    return messages
```

这段代码的意义是：

- 普通对话和结构化输出都要复用同一套消息构造逻辑
- 避免把 `system + history + user` 这套流程到处复制

这就是很典型的“把公共逻辑抽成函数”。

### 代码 2：保存当前对话

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def _save_turn(self, user_input: str, answer: str) -> None:
    self.history.append({"role": "user", "content": user_input})
    self.history.append({"role": "assistant", "content": answer})
```

这段代码的作用：

- 统一保存对话历史
- 让普通聊天和结构化输出都能保留上下文

### 代码 3：生成结构化学习计划

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def create_study_plan(self, topic: str) -> dict:
    user_input = f"""
请围绕主题“{topic}”生成一个 3 天学习计划，并严格只返回 JSON。
...
""".strip()

    response = self.client.chat.completions.create(
        model=self.settings.model,
        messages=self._build_messages(user_input),
        temperature=0.3,
    )
```

这一段做了三件事：

1. 明确要求模型输出 JSON
2. 给出固定字段格式
3. 把温度调低，提高输出稳定性

这里你要理解一个经验：

**结构化输出通常比普通聊天更适合用较低温度。**

因为我们的目标不是“更有创意”，而是“更稳定”。

### 代码 4：把模型输出解析成 Python 对象

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
raw_answer = response.choices[0].message.content or ""
cleaned_answer = self._strip_json_wrapper(raw_answer)
plan = json.loads(cleaned_answer)
```

这一步是本节最关键的工程动作。

它说明：

- 模型输出先是字符串
- 程序再把字符串变成真正的 Python `dict`

从这里开始，你的 Agent 输出就可以被程序继续消费了。

### 代码 5：命令入口

文件：[src/main.py](/mnt/d/AIcodes/Agent/src/main.py)

```python
if user_input.startswith("/plan "):
    topic = user_input[6:].strip()
    ...
    plan = agent.create_study_plan(topic)
    print(json.dumps(plan, ensure_ascii=False, indent=2))

```

这段代码的作用：

- 把 `/plan` 识别成结构化任务入口
- 调用结构化输出函数
- 把 Python 对象格式化打印

## 6. 常见错误

### 错误 1：只在 Prompt 里说“请返回 JSON”，但程序端不解析

后果：

- 表面上看像结构化输出
- 实际上程序还是拿到一段字符串

### 错误 2：不限制额外解释文字

后果：

模型可能输出：

- 先解释一段
- 再贴 JSON

这样 `json.loads(...)` 很容易失败。

### 错误 3：结构化任务还用很高温度

后果：

- 字段不稳定
- 容易多字段
- 容易少字段

### 错误 4：不做输出清洗

后果：

模型有时会包一层：

```json
```json
{...}
```
```

如果你不处理这层包装，解析就会报错。

## 7. 常见面试问题

### 问题 1：为什么 Agent 需要结构化输出？

回答要点：

- 自然语言适合人读，不适合程序稳定消费
- 结构化输出是 Tool Calling、Workflow、前端渲染、评测的基础
- 它提高了系统可控性

深入追问：

- 结构化输出和 Tool Calling 有什么关系？

### 问题 2：只靠 Prompt 约束 JSON 输出够吗？

回答要点：

- 对入门项目通常够用
- 但仍然不绝对可靠
- 程序端必须做解析和校验

深入追问：

- 更严格的做法是什么？
- 什么时候应该上 JSON Schema / structured outputs API？

### 问题 3：为什么结构化输出通常要降低 temperature？

回答要点：

- 我们追求的是稳定字段，不是发散创意
- 温度越高，格式漂移风险越大

深入追问：

- 哪些任务适合高温度，哪些任务适合低温度？

### 问题 4：结构化输出失败时，程序应该怎么做？

回答要点：

- 清洗格式
- 捕获解析异常
- 重新提示模型
- 或返回兜底错误信息

深入追问：

- 如果要做生产级系统，校验层应该放在哪一层？

## 8. 本节验收

你现在可以这样测试：

```text
/plan RAG
```

理想结果是终端输出一个格式化 JSON，里面至少包含：

- `topic`
- `target_level`
- `days`
- `tips`

然后你再试：

```text
/plan LangChain
```

如果连续两次都能稳定输出合法 JSON，这一节就算真正跑通了。

## 9. 这一节和后面有什么关系

这一节其实是后面很多能力的基础层：

- Tool Calling：本质上也是结构化参数
- RAG：检索结果也常常要结构化整理
- Workflow：节点之间最好传结构化数据
- 前端展示：最适合吃 JSON

所以你现在做的不是一个“小技巧”，而是在给后面的 Agent 体系打地基。
