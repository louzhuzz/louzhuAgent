# 第二十九步：工具型学习助手

## 1. 本节目标

这一节要做的事情是：

- 不再让你手工先 `/notes` 再 `/read`
- 而是让程序根据问题自动选择一个知识点文件
- 然后读取这份文件
- 最后基于这份工具结果回答问题

这就是“工具型学习助手”的第一版。

对应文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)
- [src/main.py](/mnt/d/AIcodes/Agent/src/main.py)
- [src/prompts.py](/mnt/d/AIcodes/Agent/src/prompts.py)
- [prompts/tool_learning_assistant_prompt.txt](/mnt/d/AIcodes/Agent/prompts/tool_learning_assistant_prompt.txt)

## 2. 是什么

这一节的重点不是“模型会不会答题”，而是：

**程序开始根据问题主动组合工具能力。**

当前这条链路是：

1. 用户输入问题
2. 程序从 `知识点/` 文件列表里挑一个最相关的文件
3. 程序读取这个文件
4. 程序把文件内容和问题一起交给模型
5. 模型基于这份工具结果作答

这就是最小的“工具型学习助手”。

## 3. 为什么

上一节你已经有：

- 工具函数
- 工具 schema
- 参数校验

但那时还是：

- 你手工决定调用哪个工具

现在开始往 Agent 迈进一步，就要让程序自己做一层决策：

- 先选哪个知识点文件
- 再调用哪个工具

虽然当前还不是“模型自动 tool calling”，但已经具备了一个重要特征：

**系统不再只会聊天，而是会先拿工具结果，再回答。**

## 4. 怎么做

### 第一步：新增一个专用 Prompt 模板

我们新增了：

- [prompts/tool_learning_assistant_prompt.txt](/mnt/d/AIcodes/Agent/prompts/tool_learning_assistant_prompt.txt)

这个模板专门用于：

- 把“已选中的知识点文件名”
- “该文件内容”
- “用户问题”

组织成一条清晰的任务输入。

为什么这里不直接写死在 `agent.py` 里？

因为这一步其实已经是一个独立业务场景：

- 基于工具结果回答

既然是独立场景，就应该有独立模板。

### 第二步：让 Agent 能从问题里提取关键词

在 [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py) 中，我们新增了：

- `_extract_query_terms(text)`

这个函数会从问题里提取：

- 英文词
- 中文短语
- 中文二元片段

它的目的不是做复杂 NLP，而是先给“选择知识点文件”提供一个最小可用的信号。

### 第三步：根据文件名给知识点打分

我们新增了：

- `_score_note_name(question, file_name)`

这一步做的是：

- 看问题关键词和文件名有多少重叠

重叠越多，说明这个文件越可能和问题相关。

### 第四步：自动选出最相关文件

我们新增了：

- `_select_note_for_question(question)`

这一步的数据流是：

1. 调 `list_notes_tool()` 列出全部知识点文件
2. 对每个文件名计算分数
3. 按分数排序
4. 选出分数最高的一篇

### 第五步：把“选文件 + 读文件 + 回答”封装成一个方法

我们新增了：

- `answer_with_note_tool(question)`

它会一次性完成：

1. 选文件
2. 读文件
3. 构造 Prompt
4. 调模型回答

### 第六步：给命令行新增 `/study`

现在你可以直接输入：

```text
/study 什么是输出解析器？
```

系统会自动：

- 选择类似 `13-输出解析器.md` 这样的文件
- 再基于该文件回答

## 5. 关键代码

### 代码 1：基于工具结果回答的 Prompt

文件：[prompts/tool_learning_assistant_prompt.txt](/mnt/d/AIcodes/Agent/prompts/tool_learning_assistant_prompt.txt)

```text
参考笔记文件名：
{file_name}

参考笔记内容：
{note_content}

用户问题：
{question}
```

这一段的意义是：

- 让模型知道“这份内容不是它自己想出来的”
- 而是程序通过工具读出来的

这对后面做更真实的 Tool Calling 很重要，因为你需要让模型清楚地区分：

- 自己生成的内容
- 工具返回的内容

### 代码 2：提取问题关键词

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def _extract_query_terms(self, text: str) -> set[str]:
    english_terms = set(re.findall(r"[a-z0-9]+", text.lower()))
    chinese_blocks = re.findall(r"[\u4e00-\u9fff]+", text)

    chinese_terms: set[str] = set()
    for block in chinese_blocks:
        cleaned = block.strip()
        if len(cleaned) >= 2:
            chinese_terms.add(cleaned)
        for index in range(len(cleaned) - 1):
            chinese_terms.add(cleaned[index : index + 2])
```

你要注意，这里不是在做“高精度语义检索”，而是在做一个够用的文件名匹配信号。

为什么这里要同时保留：

- 整段中文短语
- 中文二元片段

因为文件名往往比较短。

如果你只保留整句问题，文件名命中率会很低；补上二元片段后，像“输出”“解析”“解析器”这类片段更容易和文件名重合。

### 代码 3：给文件名打分

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def _score_note_name(self, question: str, file_name: str) -> float:
    query_terms = self._extract_query_terms(question)
    if not query_terms:
        return 0.0

    lowered_name = file_name.lower()
    hit_count = sum(1 for term in query_terms if term in lowered_name)
    return hit_count / len(query_terms)
```

这一步是在回答一个非常具体的问题：

**“这个文件名和当前问题，有多像？”**

这里为什么返回的是“命中比例”而不是“命中数量”？

因为不同问题长度不同。

如果只看命中数量，长问题和短问题的分数会不公平；用比例会更稳定一点。

### 代码 4：自动选文件

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def _select_note_for_question(self, question: str) -> str:
    note_names = self.list_notes_tool()
    if not note_names:
        raise ValueError("当前没有可用的知识点文件。")

    scored_notes = sorted(
        note_names,
        key=lambda file_name: (self._score_note_name(question, file_name), file_name),
        reverse=True,
    )
    return scored_notes[0]
```

这段代码说明：

- 决策者是程序
- 不是模型

当前版本的决策依据很简单：

- 文件名匹配分数

后面如果升级，可以换成：

- 先召回文件
- 再用模型或 rerank 排序

但当前这一节先把最小流程跑通最重要。

### 代码 5：工具型学习助手主流程

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def answer_with_note_tool(self, question: str) -> tuple[str, str]:
    file_name = self._select_note_for_question(question)
    note_content = self.read_note_tool(file_name)
    user_input = render_tool_learning_prompt(file_name, note_content, question)
    answer = self._invoke_text(user_input, temperature=0.3)
    self._save_turn(question, answer)
    return file_name, answer
```

这段代码就是这节课最核心的数据流。

按顺序看：

1. `file_name = self._select_note_for_question(question)`
   作用：先决定读哪个文件

2. `note_content = self.read_note_tool(file_name)`
   作用：调用真实工具去读内容

3. `user_input = render_tool_learning_prompt(...)`
   作用：把工具结果包装成给模型的任务输入

4. `answer = self._invoke_text(...)`
   作用：让模型基于工具结果回答

也就是说：

**这已经不是“直接问模型”，而是“先用工具拿资料，再问模型”。**

## 6. 常见错误

### 错误 1：以为这已经等于完整 Tool Calling Agent

还不是。

当前版本仍然是：

- 程序先决定工具怎么用

不是：

- 模型自己返回 tool call 请求

### 错误 2：文件选择只靠文件名，就以为足够了

这只是第一版。

它适合教学，但在更复杂项目里往往不够，因为：

- 文件名可能不完整
- 问题可能和文件标题不完全重合

### 错误 3：读到文件后，仍然像普通聊天一样随便回答

这会浪费工具价值。

所以我们专门用了独立 Prompt，明确要求模型基于这份笔记回答。

### 错误 4：把工具结果和用户问题混在一起，不做清晰分层

后果是模型不容易分清：

- 哪部分是参考资料
- 哪部分是用户真正的问题

## 7. 常见面试问题

### 问题 1：工具型学习助手和普通聊天助手的核心差别是什么？

回答要点：

- 普通聊天主要依赖模型参数知识
- 工具型学习助手会先调用外部能力拿资料
- 再基于工具结果回答

### 问题 2：为什么这一节先让程序决定“读哪个文件”，而不是让模型决定？

回答要点：

- 先把数据流和边界学清楚
- 程序控制更稳定，便于调试
- 后面再升级到模型自动调用工具会更顺

### 问题 3：这套文件选择逻辑的局限是什么？

回答要点：

- 只看文件名，不看正文语义
- 对表述差异较大的问题不够鲁棒
- 后面可以升级成检索、rerank 或模型选择
