# 第六步：让 Agent 拥有多轮对话记忆

## 1. 本节目标

这一节要把你的 Agent 从：

- 单轮问答

升级成：

- 能记住本轮会话上下文的多轮 Agent

并且支持：

- `/clear` 清空当前记忆

## 2. 是什么

多轮对话记忆，指的是：

Agent 在回答当前问题时，会把之前几轮对话也一起作为上下文传给模型。

注意：

这不是模型“自动记住了”。

而是你的程序把历史消息保存下来，并在下一次请求时重新发送给模型。

## 3. 为什么

如果没有多轮记忆，Agent 就只能：

- 问一句
- 答一句
- 然后忘掉

这样它很难处理这些场景：

- 连续追问
- 基于上一轮继续展开
- 逐步完成一个学习任务

所以多轮记忆是 Agent 从“聊天接口”升级到“有状态系统”的第一步。

## 4. 怎么做

### 第一步：在 Agent 类里增加历史记录容器

最简单的方法就是放一个列表：

- 保存用户消息
- 保存模型回复

### 第二步：每次调用模型前，把历史消息拼进去

新的请求消息结构变成：

1. system prompt
2. 历史消息
3. 当前用户输入

### 第三步：模型回答后，把这轮问答存起来

这样下一轮才能继续带上。

### 第四步：增加清空命令

让你可以主动清空上下文，方便：

- 调试
- 对比效果
- 避免旧上下文污染新问题

## 5. 关键代码

### 代码 1：初始化历史记录

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
class LearningAgent:
    def __init__(self, settings: Settings):
        self.settings = settings
        self.client = OpenAI(
            api_key=settings.api_key,
            base_url=settings.base_url,
        )
        self.system_prompt = load_system_prompt()
        self.history: list[dict[str, str]] = []
```

这段代码做了什么：

- 增加 `self.history`
- 用列表存放历史消息

这里的结构和 OpenAI 兼容消息格式保持一致：

```python
{"role": "user", "content": "..."}
{"role": "assistant", "content": "..."}
```

### 代码 2：把历史消息带入模型请求

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def reply(self, user_input: str) -> str:
    messages = [{"role": "system", "content": self.system_prompt}, *self.history]
    messages.append({"role": "user", "content": user_input})
```

这一步是本节最核心的代码。

作用是：

- 先放 system prompt
- 再拼接历史消息
- 最后放当前用户输入

你要抓住一个本质：

**模型每次看到的是这一整个 `messages` 列表。**

不是“模型自己记得前面说过什么”。

### 代码 3：保存本轮对话

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
answer = response.choices[0].message.content or ""
self.history.append({"role": "user", "content": user_input})
self.history.append({"role": "assistant", "content": answer})
return answer
```

这一步做了两件事：

- 保存用户本轮输入
- 保存模型本轮回复

只有保存之后，下一轮才有“记忆”。

### 代码 4：清空记忆

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
def clear_history(self) -> None:
    self.history.clear()
```

这个实现非常简单，但非常重要。

它让你能主动控制上下文边界。

### 代码 5：入口命令处理

文件：[src/main.py](/mnt/d/AIcodes/Agent/src/main.py)

```python
if user_input == "/clear":
    agent.clear_history()
    print("已清空当前对话记忆。")
    continue
```

这段代码把“记忆控制能力”暴露给用户。

这就是一个很典型的工程思路：

- 能力在 `agent.py`
- 命令入口在 `main.py`

## 6. 常见错误

### 错误 1：以为模型天然会记住前文

这是最常见误解。

更准确的说法是：

模型只看你本次请求传进去的上下文。

### 错误 2：只保存用户消息，不保存助手消息

后果：

模型下一轮只知道你问过什么，不知道它自己答过什么。

这会让多轮对话很容易断裂。

### 错误 3：历史无限增长

后果：

- 成本变高
- 速度变慢
- 旧上下文干扰当前任务

这也是后面为什么要学：

- 上下文裁剪
- 对话总结
- 长期记忆和短期记忆分层

## 7. 常见面试问题

### 问题 1：多轮对话记忆是怎么实现的？

回答要点：

- 程序保存历史消息
- 每次请求把 `system + history + current_user_input` 一起传给模型
- 模型并没有脱离请求独立记忆上下文

深入追问：

- 这和数据库持久化记忆有什么区别？

### 问题 2：为什么要同时保存 user 和 assistant 的消息？

回答要点：

- 对话是双边上下文
- 只保存用户问题，无法还原完整语义链
- 模型下一轮需要知道自己说过什么

深入追问：

- 如果只保留摘要，不保留原始消息，会有什么权衡？

### 问题 3：为什么要提供 `/clear`？

回答要点：

- 控制上下文污染
- 方便测试记忆效果
- 为后续上下文管理打基础

深入追问：

- 生产环境一般怎么做上下文裁剪，而不是完全清空？

### 问题 4：短期记忆和长期记忆有什么区别？

回答要点：

- 短期记忆：当前会话内消息上下文
- 长期记忆：跨会话保存的用户偏好、历史目标、重要事实

深入追问：

- 什么信息适合写入长期记忆？什么不适合？

## 8. 本节验收

你可以直接这样测试：

1. 输入：`我接下来想学 RAG`
2. 再输入：`给我一个 3 天计划`
3. 输入：`/clear`
4. 再输入：`给我一个 3 天计划`

如果清空前后结果明显不同，说明记忆生效了。

如果你还能自己解释清楚下面这句话，这一节就算真的理解了：

**多轮对话记忆的本质，不是模型自己记住了历史，而是程序把历史消息再次传给模型。**
