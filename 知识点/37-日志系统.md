# 第 37 课：日志系统

## 1. 本节目标

给主项目补上第一版真正可用的本地日志系统，重点记录：

- 用户输入
- 路由结果
- 知识点问答执行情况
- 工具调用
- Agent / ReAct 的步数
- 错误信息

这一课的目标不是一上来就接外部平台，而是先让主项目具备：

- 可回溯
- 可排错
- 可统计

## 2. 是什么

这一步新增的是：

- 本地 JSONL 运行日志

日志文件默认写到：

- `data/logs/agent_runtime.jsonl`

每一行都是一条结构化 JSON 事件，方便后面继续做：

- 指标统计
- 问题回放
- LangSmith / tracing 对接

## 3. 为什么

前面你的主项目已经有：

- `学习计划`
- `知识点问答`
- `任务拆解`
- `Tool Calling`
- `ReAct`
- `Agent v1`

但如果没有日志，你后面很难回答这些问题：

- 为什么这次路由错了
- 哪次知识问答没命中正确资料
- 哪个工具调用失败最多
- `/agent` 和 `/react` 平均走了几步

所以这一课的意义是：

**让主项目从“会运行”升级到“可观察”。**

## 4. 怎么做

### 第一步：新增本地日志模块

文件：

- [src/runtime_logger.py](/mnt/d/AIcodes/Agent/src/runtime_logger.py)

这个模块只做一件事：

- 把运行事件按 JSONL 形式写到本地文件

### 第二步：给配置增加日志目录

文件：

- [src/config.py](/mnt/d/AIcodes/Agent/src/config.py)
- [.env.example](/mnt/d/AIcodes/Agent/.env.example)

新增配置：

```env
LOG_DIRECTORY=data/logs
```

### 第三步：把运行层接入日志

文件：

- [src/app_runtime.py](/mnt/d/AIcodes/Agent/src/app_runtime.py)

因为命令行端和网页端都已经走统一运行层，所以最适合在这里打日志。

这样能保证：

- CLI 走的流程会记日志
- Web 走的流程也会记日志

### 第四步：命令行和网页端都共用同一个 logger

文件：

- [src/main.py](/mnt/d/AIcodes/Agent/src/main.py)
- [src/web_app.py](/mnt/d/AIcodes/Agent/src/web_app.py)

这样后面你在浏览器里问问题，和在终端里问问题，都会写进同一套日志体系里。

## 5. 关键代码

### 代码 1：最小 JSONL Logger

文件：

- [src/runtime_logger.py](/mnt/d/AIcodes/Agent/src/runtime_logger.py)

```python
record = {
    "timestamp": datetime.now(timezone.utc).isoformat(),
    "session_id": session_id,
    "event_type": event_type,
    "payload": payload,
}
```

这里每次不是写一大段字符串，而是写一个标准结构：

- `timestamp`
- `session_id`
- `event_type`
- `payload`

这会让后面继续做统计时更简单。

### 代码 2：统一运行层里的 `_log(...)`

文件：

- [src/app_runtime.py](/mnt/d/AIcodes/Agent/src/app_runtime.py)

```python
def _log(
    logger: RuntimeLogger | None,
    session_id: str,
    event_type: str,
    payload: dict[str, Any],
) -> None:
```

这里专门抽一个 `_log(...)` 小函数，是为了避免：

- 到处写 `if logger is not None`

它的工程意义是：

- 统一日志入口
- 降低重复代码
- 后面更容易替换成更复杂的日志后端

### 代码 3：在用户输入刚进入系统时记录日志

```python
_log(
    logger,
    session_id,
    "user_input_received",
    {"raw_input": user_input, "cleaned_input": cleaned},
)
```

这是非常关键的一层。

因为很多问题都要从这里排：

- 用户到底输入了什么
- 清洗后的输入是什么
- 路由前的数据长什么样

### 代码 4：在关键能力完成后记录结构化结果

例如知识点问答：

```python
_log(
    logger,
    session_id,
    "knowledge_qa_completed",
    {
        "question": question,
        "selected_notes": result["selected_notes"],
        "retrieved_chunk_count": len(result["retrieved_chunks"]),
        "index_statuses": result["index_statuses"],
    },
)
```

这里记录的不是完整回答全文，而是：

- 问了什么
- 选了哪些知识点文件
- 命中了几个 chunk
- 这些文件是 `cache_hit` 还是 `reindexed`

这类日志最适合后面做排查和指标统计。

### 代码 5：CLI 和 Web 端都共用一个 logger

```python
logger = RuntimeLogger(settings.log_directory)
result = handle_user_input(agent, user_input, logger=logger, session_id="cli")
```

以及：

```python
result = handle_user_input(agent, payload.message, logger=logger, session_id=session_id)
```

这里最关键的理解是：

**日志不是只给某一个入口用，而是给整个主项目的运行层用。**

## 6. 常见错误

### 错误 1：只打印字符串，不做结构化日志

这会导致后面很难做：

- 统计
- 检索
- 事件聚合

### 错误 2：只在 CLI 里打日志

如果网页端和 CLI 端不统一，后面会出现：

- 两边行为不一致
- 一边能排查，一边不能

### 错误 3：日志里只记最终答案，不记中间过程

后面你真正需要排查的通常是：

- 路由错了
- 工具没调到
- 检索候选错了

所以必须记中间事件。

## 7. 常见面试问题

### 问题 1：为什么 Agent 项目不能只看最终回答？

回答要点：

- Agent 是多步系统
- 最终回答错误，问题可能出在路由、检索、工具、Prompt 任意一层
- 所以必须记录中间事件

### 问题 2：为什么这里先做本地 JSONL，而不是一开始就接 LangSmith？

回答要点：

- 先把本地结构化事件打通，能快速验证日志设计是否合理
- 本地日志更简单，适合当前阶段
- 后面接 LangSmith 时，会更清楚自己到底想观察什么

### 问题 3：日志系统给后面哪些能力打基础？

回答要点：

- LangSmith / tracing
- 指标统计
- 评测体系
- 调用成功率分析
- 成本分析


```text
/v1 我要学习通信原理
/qa 什么是输出解析器？
/agent 现在有哪些知识点文件？
```

然后查看：

```text
data/logs/agent_runtime.jsonl
```

如果你能看到：

- `user_input_received`
- `agent_v1_completed`
- `knowledge_qa_completed`
- `tool_agent_completed`

说明这节已经跑通。
