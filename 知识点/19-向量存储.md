# 第十九步：向量存储

## 1. 本节目标

这一节要把前面得到的：

- chunk
- 向量

真正组织成一个“可查询的数据结构”。

本节新增的示例文件是：

- [src/vector_store_intro.py](/mnt/d/AIcodes/Agent/src/vector_store_intro.py)

## 2. 是什么

向量存储，可以先理解成：

**专门用来保存“文档 + 向量”并支持相似度查询的一层。**

在真实 RAG 系统里，这一层通常会是：

- Chroma
- FAISS
- Milvus
- pgvector

但在这一节里，我们先做一个：

**内存版最小向量库。**

## 3. 为什么

前一课你已经能：

- 把文本变成向量
- 算两个向量的相似度

但如果没有“存储层”，这些能力还是散的。

你需要有一个统一结构，把：

- 原始 chunk
- 对应向量

一起保存起来，并且支持：

- `add_documents(...)`
- `similarity_search(...)`

这就是向量存储层的意义。

## 6. 常见错误

### 错误 1：只存向量，不存原文档

后果：

- 找到了相似结果，却拿不到原始内容和 metadata

### 错误 2：把向量存储和 embedding 混成一件事

实际上：

- embedding 负责把文本变成向量
- 向量存储负责保存并检索这些向量

### 错误 3：以为内存版向量库就等于真实生产方案

不是。

这节的重点是理解抽象层，不是替代真实数据库。

## 7. 常见面试问题

### 问题 1：向量存储的核心职责是什么？

回答要点：

- 保存文档及其向量表示
- 支持相似度检索
- 为 Retriever 提供底层能力

深入追问：

- 为什么说 embedding 和 vector store 是两层不同职责？

### 问题 2：为什么向量库里不能只存向量？

回答要点：

- 检索结果最终要回到原文内容
- 还需要 metadata 来做引用和追溯
- 只存向量无法直接支撑后续回答生成

深入追问：

- 真实系统里通常还会额外存哪些字段？

### 问题 3：内存版向量库和真实向量数据库差别在哪里？

回答要点：

- 内存版简单、易理解、适合教学
- 真实向量库支持持久化、索引优化、规模扩展
- 真实系统还会考虑更新、删除、过滤查询等问题

深入追问：

- 为什么大规模场景不能只靠 Python 列表遍历？

### 问题 4：为什么向量存储是 Retriever 的前置层？

回答要点：

- Retriever 需要一个可查询的向量底座
- 向量存储负责“存和查”
- Retriever 负责把检索逻辑抽象成更高层能力

深入追问：

- 向量存储和 Retriever 的边界应该怎么理解？
