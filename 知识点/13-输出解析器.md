# 第十三步：输出解析器

## 1. 本节目标

这一节要把项目从：

- 模型返回后，业务代码自己顺手处理结果

升级成：

- 模型返回后，先进入专门的解析层

本节完成后，你会得到一个独立文件：

- [src/output_parsers.py](/mnt/d/AIcodes/Agent/src/output_parsers.py)

它专门负责：

- 文本解析
- JSON 清洗
- JSON 解析
- 解析错误抛出

## 2. 是什么

输出解析器，本质上就是：

**把模型原始输出转换成业务真正需要的数据格式的一层代码。**

在当前项目里，最典型的两类输出是：

1. 普通文本输出
2. 结构化 JSON 输出

所以本节我们拆出了两个解析动作：

- `parse_text_output(...)`
- `parse_json_output(...)`

## 3. 为什么

如果解析逻辑散落在业务代码里，会出现几个问题：

- 解析规则到处重复
- 错误处理不统一
- 后面很难扩成更复杂的 Parser

把解析层单独抽出来后，好处是：

- `agent.py` 更聚焦业务流程
- 解析规则可复用
- 后面接 LangChain Output Parser 时更容易理解

所以这一步不是单纯“整理代码”，而是在建立：

**模型输出层 -> 解析层 -> 业务层**

这条更清晰的边界。

## 4. 怎么做

### 第一步：抽出文本解析

模型返回的内容不一定永远是最简单的字符串。

所以我们把：

- 响应内容转文本

这一步统一放进：

- `parse_text_output(...)`

### 第二步：抽出 JSON 清洗与解析

结构化输出时，模型有可能返回：

- 纯 JSON
- 带代码块包裹的 JSON

所以解析流程应分成两步：

1. `strip_code_fence(...)`
2. `parse_json_output(...)`

### 第三步：统一抛解析错误

如果解析失败，不应该让错误表现得很模糊。

所以我们新增了：

- `OutputParserError`

它的价值在于：

- 让错误类型更清楚
- 方便后续做统一异常处理

### 第四步：让业务代码只负责调用解析器

现在 `agent.py` 不再亲自写：

- `json.loads(...)`
- 去代码块逻辑
- 文本拼接逻辑

而是交给解析器模块处理。

## 5. 关键代码

### 代码 1：文本解析器

文件：[src/output_parsers.py](/mnt/d/AIcodes/Agent/src/output_parsers.py)

```python
def parse_text_output(content: object) -> str:
    if isinstance(content, str): # isinstance
        return content
    ...
```

这段代码的作用：

- 统一把模型响应内容转成字符串
- 不让业务层假设“返回值永远是 str”

### 代码 2：代码块清洗

文件：[src/output_parsers.py](/mnt/d/AIcodes/Agent/src/output_parsers.py)

```python
def strip_code_fence(text: str) -> str:
    cleaned = text.strip()
    if cleaned.startswith("```json"):
        cleaned = cleaned[7:]
```

这段代码的作用：

- 去掉模型常见的 Markdown 包装
- 让后续 JSON 解析更稳定

### 代码 3：JSON 解析器

文件：[src/output_parsers.py](/mnt/d/AIcodes/Agent/src/output_parsers.py)

```python
def parse_json_output(raw_text: str) -> dict[str, Any]:
    cleaned = strip_code_fence(raw_text)
    parsed = json.loads(cleaned)
```

这一段是本节最关键的解析动作。

它说明：

- 模型输出先是字符串
- 清洗后再转成 Python 对象

### 代码 4：主 Agent 接入解析层

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
return parse_text_output(response.content)
```

和：

```python
plan = parse_json_output(raw_answer)
```

这两处接入说明：

- 普通聊天走文本解析器
- 结构化输出走 JSON 解析器

这就是“解析层按输出类型分流”。

## 6. 常见错误

### 错误 1：在业务代码里到处写 `json.loads(...)`

后果：

- 解析逻辑重复
- 错误处理混乱

### 错误 2：默认认为模型一定返回合法 JSON

后果：

- 一旦模型多返回一句解释文字，程序就会崩

### 错误 3：不区分文本解析和 JSON 解析

后果：

- 所有输出都按一种方式处理
- 后面扩展 Parser 很难

## 7. 常见面试问题

### 问题 1：为什么要单独做输出解析层？

回答要点：

- 分离业务逻辑和输出处理逻辑
- 提高复用性
- 提高错误处理一致性

深入追问：

- 如果没有解析层，项目后面会在哪些地方开始失控？

### 问题 2：为什么结构化输出不能只靠模型保证格式？

回答要点：

- 模型是概率生成，不是强类型系统
- 程序必须做二次校验
- 解析器就是“模型输出到程序对象”的守门员

深入追问：

- 更严格的下一步是什么？Schema 校验还是 Pydantic？

### 问题 3：为什么要定义 `OutputParserError`？

回答要点：

- 让解析失败更可识别
- 便于统一异常处理
- 为后续更复杂解析链做准备

深入追问：

- 如果要做生产级 Agent，解析错误应该怎么上报和记录？

### 问题 4：输出解析器和 LangChain Output Parser 有什么关系？

回答要点：

- 思想一致，都是“把模型输出转成可消费对象”
- 现在是手写版，帮助理解底层
- 后面学习 LangChain Output Parser 时会更容易映射

深入追问：

- 什么时候应该从手写解析器迁移到 LangChain Parser？
