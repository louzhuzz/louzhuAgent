# 第五步：把模型接到火山方舟

## 这一改动的核心结论

你现在这个项目，**可以继续使用 `openai` Python SDK**，不用因为切到火山方舟就重写一套调用逻辑。

原因是火山方舟官方文档明确说明：

- 模型调用 API 与 OpenAI API 协议兼容
- 可以使用兼容 OpenAI API 的第三方 SDK
- 只需要调整 `base_url`、`api_key`、`model`

## 当前我们为什么这样改

我们现在的目标不是研究 SDK 差异，而是先理解 Agent 主链路。

如果为了平台切换就把代码结构全换掉，你会把注意力浪费在：

- SDK 用法差异
- 文档切换
- 参数细节

而真正重要的东西其实没变：

- 输入消息
- 系统提示词
- 模型调用
- 输出结果

所以当前阶段最合理的做法是：

**保留调用结构，只替换配置层。**

## 现在 `.env` 里每个字段是什么意思

### `ARK_API_KEY`

你的火山方舟 API Key。

### `ARK_BASE_URL`

模型调用入口。

当前默认写的是：

`https://ark.cn-beijing.volces.com/api/v3`

### `ARK_MODEL`

你要调用的模型标识。

根据火山方舟官方文档和控制台流程，实际使用时通常要填：

- 你的 `Endpoint ID`
- 或平台当前要求的模型名

最稳妥的做法是：

**直接以你在火山方舟控制台可调用的标识为准。**

## 关键知识点 1：为什么 `base_url` 这么重要

`base_url` 决定你的请求被发到哪个平台。

同样一份 `chat.completions.create(...)` 代码：

- 如果发到 OpenAI，就走 OpenAI
- 如果发到火山方舟，就走火山方舟

所以很多“接平台”的本质，不是重写业务逻辑，而是：

**替换调用入口和鉴权参数。**

## 关键知识点 2：为什么先保留 `chat.completions`

因为你现在还在做最小 Agent。

你最需要先理解的是：

- `system` 和 `user` 消息怎么组织
- 模型怎么返回结果
- 主程序怎么把结果打印出来

等你这套跑稳了，我们再决定要不要升级到：

- `responses` API
- 工具调用
- 结构化输出

## 关键知识点 3：火山方舟和 OpenAI 在这一步的共同抽象

你可以先抽象成：

`统一消息格式 + 不同平台配置`

这就是为什么我们现在的项目结构没必要大改。

## 你下一步要做的事

1. 去火山方舟控制台创建或确认可用的 API Key
2. 确认可调用的模型标识
3. 把 `.env.example` 复制成 `.env`
4. 填入真实配置
5. 运行 `python src/main.py`
