## 从现在开始的文档标准

后续每一篇 `知识点/*.md` 都不再只讲：

- 是什么
- 为什么

还必须补齐这三类内容：

1. 怎么做
2. 关键代码
3. 常见面试问题

也就是说，后面的每篇文档默认都按下面这个模板来写：

### 1. 本节目标

这一步要实现什么功能。

### 2. 是什么

解释概念定义。

### 3. 为什么

解释这一层在 Agent 体系里的作用。

### 4. 怎么做

给出具体实现步骤，最好能直接照着操作。

### 5. 关键代码

贴当前项目里的核心代码片段，并解释每一段代码在做什么。

### 6. 常见错误

列出最容易踩坑的点和排查方式。

### 7. 常见面试问题

不只给标准答案，还要强调深入理解点和追问方向。

### 8. 本节验收

告诉你怎么验证这一节真的学会了、功能真的生效了。

## 项目化之前的版本基线

这一段是现在必须补上的，因为你已经明确要求：

- LangChain 要用 `1.0+`

而且当前官方主线也确实已经切到了 `1.x`。

### 为什么现在就要定版本

因为你后面要进入的是：

- 主项目长期迭代
- 不是一次性示例脚本

如果项目化之前不先统一版本基线，后面会出现这些问题：

- 教学文档和实际代码版本不一致
- 旧 API 和新 API 混用
- 学的是一套，项目里用的是另一套

### 当前建议

项目化之前，统一采用：

- `LangChain 1.x`
- `langchain-openai 1.x`

也就是说，后面真正进入主项目工程化时，默认思路要逐渐靠近：

- `langchain.agents.create_agent(...)`
- `LangChain v1 agents`
- `LangSmith`
- `LangGraph`

### 这对当前大纲意味着什么

它意味着：

1. 前面已经做过的很多 `Runnable / Parser / 手写 Agent Loop` 课程仍然保留  
   因为它们对理解底层非常有价值

2. 但后面进入“项目化实战”时，不应再继续把旧风格抽象当成最终目标  
   而要逐步切到 `LangChain 1.x` 的推荐主线

3. 后面再讲 `LangGraph` 时，也要明确：

- `LangChain v1` 的 agent 本来就是构建在 `LangGraph` 之上
- 学 `LangGraph` 是为了更强控制力，不是因为 `LangChain agent` 不够“正统”

### 一句话原则

**前面的课保留“为了理解底层”，后面的项目化改成“为了贴近 LangChain 1.x 官方主线”。**

## 按 LangChain 1.x 重新理解后半段主线

如果只看你已经完成到现在的位置，并且考虑你要求后面统一到 `LangChain 1.0+`，那么后半段最合理的推进方式应该是：

1. 先完成当前“手写 Agent / Tool Calling / ReAct”主线  
   目的是把底层机制吃透

2. 在进入项目化之前，统一依赖版本和目录结构  
   目的是让主项目和旧示例彻底分层

3. 项目化阶段开始构建真正的主项目功能  
   例如：
   - 学习计划生成器
   - 知识点问答系统
   - 任务拆解器
   - 学习助理 Agent v1

4. 完成主项目第一版之后，再做一次 LangChain 1.x 重构  
   重点是：
   - `create_agent`
   - 1.x agent 组织方式
   - 观察 LangSmith 中的运行链路

5. 当你开始需要更细粒度的状态控制和工作流时，再进入 LangGraph  
   这样顺序会比“先学图框架再做项目”更自然
