# 第三十二步：学习计划生成器（项目化重构版）

## 1. 本节目标

这一节不是重新做一遍早期的 `/plan` 示例，而是把它升级成：

- 主项目里的正式模块
- 有输入对象
- 有服务层
- 有结果校验

也就是说，从这一节开始，学习计划功能不再只是“某个方法里顺手写的一段 Prompt 调用”，而是开始具备真正的项目结构。

对应文件：

- [src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)
- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)
- [src/main.py](/mnt/d/AIcodes/Agent/src/main.py)
- [src/prompts.py](/mnt/d/AIcodes/Agent/src/prompts.py)
- [prompts/study_plan_prompt.txt](/mnt/d/AIcodes/Agent/prompts/study_plan_prompt.txt)

## 2. 是什么

这一节做的核心升级有三层：

1. 把学习计划请求抽成独立对象
2. 把生成逻辑抽成服务层
3. 把命令行输入解析和结果校验独立出来

以前的教学版思路是：

- `/plan topic`
- 直接拼 Prompt
- 直接让模型回 JSON

现在的项目化思路是：

- 先把用户输入解析成 `StudyPlanRequest`
- 再交给 `StudyPlanService`
- 服务层负责：
  - 校验输入
  - 生成 Prompt
  - 调模型
  - 校验输出

## 3. 为什么

如果继续沿用早期 demo 式写法，会出现几个问题：

- `/plan` 只能传一个主题，表达能力太弱
- Prompt、解析、校验全堆在 `agent.py`
- 后面要扩展成更完整的主项目功能会越来越乱

而项目化重构的目标就是：

**让一个功能不只是“能跑”，还要“可扩展、可维护、可继续演进”。**

## 4. 怎么做

### 第一步：定义请求对象

我们新增了：

- [src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)

里面的 `StudyPlanRequest` 用来描述一次学习计划生成请求。

它目前包含：

- `topic`
- `current_level`
- `days`
- `goal`

这意味着你的学习计划功能不再只知道“主题是什么”，还知道：

- 用户现在基础如何
- 希望学几天
- 想达到什么目标

### 第二步：新增服务层

我们新增了：

- `StudyPlanService`

它负责：

1. 校验请求
2. 生成 Prompt
3. 调模型拿 JSON
4. 校验返回结果

这样 `LearningAgent` 不需要再亲自处理所有细节。

### 第三步：让 `/plan` 支持更完整的输入

现在的格式变成：

```text
/plan LangChain | 零基础 | 5 | 做出一个可运行 Demo
```

当然，为了兼容老用法，这种也可以：

```text
/plan LangChain
```

这时系统会自动补默认值。

### 第四步：加强 Prompt

新的 [prompts/study_plan_prompt.txt](/mnt/d/AIcodes/Agent/prompts/study_plan_prompt.txt) 不再只看主题，而是会把：

- 当前基础
- 学习天数
- 学习目标

一起交给模型。

### 第五步：增加返回结果校验

现在不只是“模型返回 JSON 就算完”，还会检查：

- `days` 是否真的是请求要求的天数

这一步很关键，因为主项目功能不能完全信任模型输出。

## 5. 关键代码

### 代码 1：学习计划请求对象

文件：[src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)

```python
@dataclass
class StudyPlanRequest:
    topic: str
    current_level: str = "零基础"
    days: int = 3
    goal: str = "完成系统入门并输出可验证的学习成果"
```

这段代码的价值在于：

- 把“散落的参数”收拢成一个明确对象

这比把 4 个字符串参数到处传来传去更稳定，也更容易扩展。

### 代码 2：命令解析器

文件：[src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)

```python
def parse_study_plan_request(raw_text: str) -> StudyPlanRequest:
    parts = [part.strip() for part in raw_text.split("|")]
```

这一步是把命令行字符串变成项目内部对象。

为什么这一步要单独抽出来？

因为：

- 命令行输入格式属于“入口层逻辑”
- 不应该和模型调用逻辑混在一起

### 代码 3：服务层

文件：[src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)

```python
class StudyPlanService:
    def __init__(
        self,
        invoke_json: Callable[[str, float], dict[str, Any]],
        render_prompt: Callable[[StudyPlanRequest], str],
    ) -> None:
        self.invoke_json = invoke_json
        self.render_prompt = render_prompt
```

这段代码的重点是“依赖注入”。

也就是说：

- `StudyPlanService` 不自己创建模型
- 它只依赖一个“能返回 JSON 的调用函数”
- 以及一个“能渲染 Prompt 的函数”

这样它就不会和某个具体模型写死耦合。

### 代码 4：结果校验

文件：[src/study_plan.py](/mnt/d/AIcodes/Agent/src/study_plan.py)

```python
def _validate_plan(self, plan: dict[str, Any], request: StudyPlanRequest) -> dict[str, Any]:
    days = plan.get("days")
    if not isinstance(days, list):
        raise ValueError("学习计划返回结果缺少合法的 days 列表。")

    if len(days) != request.days:
        raise ValueError(
            f"模型返回的 days 数量是 {len(days)}，但请求要求 {request.days} 天。"
        )
```

这一步体现了项目化最关键的一点：

**模型输出不是最终真相，程序必须做二次校验。**

### 代码 5：主项目接入服务层

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
self.study_plan_service = StudyPlanService(
    invoke_json=self._invoke_json,
    render_prompt=render_study_plan_prompt,
)
```

这说明 `LearningAgent` 现在不再亲自写 `/plan` 的内部细节，而是把这项能力交给专门模块。

### 代码 6：新的 `/plan` 命令入口

文件：[src/main.py](/mnt/d/AIcodes/Agent/src/main.py)

```python
request = parse_study_plan_request(raw_request)
plan = agent.create_study_plan(request)
```

这两行非常能体现“项目化”：

1. 先把原始文本解析成对象
2. 再调用主项目功能模块

## 6. 常见错误

### 错误 1：以为项目化重构就是把代码拆成更多文件

不是。

真正的重点是：

- 职责有没有变清晰
- 输入输出有没有边界
- 后续扩展会不会更容易

### 错误 2：仍然把 `/plan` 当作“只有主题”一个参数

那样主项目表达能力太弱，不适合继续扩展。

### 错误 3：模型返回 JSON 就完全信任

主项目功能一定要做最基本的输出校验。

### 错误 4：把命令解析、业务逻辑、Prompt 渲染全塞回 `agent.py`

这样会把刚做好的项目边界又打乱。

## 7. 常见面试问题

### 问题 1：为什么学习计划生成功能要抽服务层？

回答要点：

- 降低 `LearningAgent` 的职责复杂度
- 让功能模块可以独立扩展和测试
- 更接近真实项目分层

### 问题 2：为什么需要请求对象 `StudyPlanRequest`？

回答要点：

- 让输入结构明确
- 提升可读性
- 后面加字段时更自然

### 问题 3：为什么主项目功能要对模型输出做二次校验？

回答要点：

- 模型是概率生成，不是强类型系统
- 主项目不能把模型输出当成绝对可信
- 校验是保障稳定性的必要手段

## 8. 本节验收

运行：

```bash
python src/main.py
```

然后测试：

```text
/plan LangChain
/plan LangChain | 零基础 | 5 | 做出一个可运行 Demo
/plan RAG | 有一些 Python 基础 | 7 | 做出一个本地知识问答系统
```

你应该观察到：

- 老写法仍然能用
- 新写法能表达更完整需求
- 返回的 `days` 数量会和请求保持一致

如果这三点都成立，这一节就算真正进入了项目化重构。
