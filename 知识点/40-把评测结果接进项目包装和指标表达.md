# 第 40 课：把评测结果接进项目包装和指标表达

## 1. 本节目标

把前一课已经跑起来的评测体系，真正接到“项目包装目标”上。

这一课要解决的是两个现实问题：

1. 你已经有评测报告了，但简历和面试里怎么讲
2. 你已经有运行日志了，但怎么把这些数据整理成能直接复用的指标

所以这一步不是继续做模型能力，而是做：

- 指标摘要层
- 项目包装话术层
- 结果落盘层

## 2. 是什么

这一步新增的是一个“项目指标摘要器”。

对应文件：

- [src/project_metrics.py](/mnt/d/AIcodes/Agent/src/project_metrics.py)
- [src/export_project_metrics.py](/mnt/d/AIcodes/Agent/src/export_project_metrics.py)

输入来源是：

- [latest_eval_report.json](/mnt/d/AIcodes/Agent/data/evals/latest_eval_report.json)
- [agent_runtime.jsonl](/mnt/d/AIcodes/Agent/data/logs/agent_runtime.jsonl)

输出结果是：

- [latest_project_metrics.json](/mnt/d/AIcodes/Agent/data/metrics/latest_project_metrics.json)
- [latest_project_metrics.md](/mnt/d/AIcodes/Agent/data/metrics/latest_project_metrics.md)

## 3. 为什么

前一课的评测体系已经回答了：

- 这次改动有没有退化

但它还没有回答：

- 这个项目现在能量化成什么指标
- 这些指标怎么转成简历语言

如果没有这一层，你后面会遇到两个问题：

1. 你知道项目“最近 5 条样例全过了”，但简历里不会写
2. 你知道系统在记日志，但说不出“当前累计了多少事件、多少次输入”

所以这一课的本质是：

**把工程数据翻译成项目表达。**

## 4. 怎么做

### 第一步：确定哪些指标最值得先抽出来

当前第一版不贪大，先抽这几类：

- 评测总样例数
- 评测通过率
- `qa` 预期笔记命中率
- `v1` 意图匹配率
- `v1` 结果类型匹配率
- 运行日志事件数
- 用户输入数

这些指标的特点是：

- 已经在当前项目里真实存在
- 计算稳定
- 很适合写进项目描述

### 第二步：从评测报告提取核心指标

文件：

- [src/project_metrics.py](/mnt/d/AIcodes/Agent/src/project_metrics.py)

这里会把：

- `total_cases`
- `passed_cases`
- `pass_rate`
- `qa` 样例的 `note_hit`
- `v1` 样例的 `intent_match`
- `v1` 样例的 `result_type_match`

整理成更适合阅读的汇总指标。

### 第三步：从运行日志提取规模类指标

同样在：

- [src/project_metrics.py](/mnt/d/AIcodes/Agent/src/project_metrics.py)

会读取：

- [agent_runtime.jsonl](/mnt/d/AIcodes/Agent/data/logs/agent_runtime.jsonl)

提取：

- 总事件数
- 会话数
- 用户输入数
- 完成类事件数
- 错误类事件数

### 第四步：自动生成“项目包装话术”

这是这一课最关键的动作。

项目里不是只生成 JSON 指标，还会自动生成类似下面这种表达：

- 构建主项目最小回归评测体系，当前固定样例 5 条，最近一次通过率 100.00%。
- 围绕知识点问答建立检索命中指标，当前 `qa` 样例的预期笔记命中率为 100.00%。
- 围绕 Agent v1 建立统一入口路由评测，当前意图匹配率为 100.00%，结果类型匹配率为 100.00%。
- 本地结构化日志已累计记录 N 条事件、M 次用户输入，便于后续继续沉淀调用规模、错误分布和性能指标。

这样后面你就不是“手工编项目成果”，而是基于项目真实数据去整理。

### 代码 2：提取评测指标

```python
def _build_eval_metrics(self, report: dict[str, Any]) -> dict[str, Any]:
    cases = report.get("cases", [])
    qa_cases = [item for item in cases if item.get("mode") == "qa"]
    v1_cases = [item for item in cases if item.get("mode") == "v1"]
```

这一段先把评测结果按模式拆开：

- `qa`
- `v1`

为什么要拆？

因为两类能力的评估重点不一样：

- `qa` 更关心文档命中和回答内容
- `v1` 更关心路由和结果类型

## 6. 常见错误

### 错误 1：只有评测报告，没有指标摘要

这样你还是得自己手工解读，很难长期复用。

### 错误 2：一上来就追求复杂业务指标

比如：

- 用户满意度
- 延迟分位数
- 工具精确召回率

这些现在项目里还没有稳定数据源。

当前更合理的是先抓：

- 通过率
- 命中率
- 路由匹配率
- 事件规模

### 错误 3：简历里硬写未经项目验证的数字

这一课的目的正好就是避免这件事。

后面你写项目成果时，应该尽量基于：

- 评测报告
- 指标摘要
- 日志统计

而不是“感觉差不多就写一个数字”。

## 7. 常见面试问题

### 问题 1：为什么有了评测体系之后，还要再做指标摘要层？

回答要点：

- 评测报告偏工程调试
- 指标摘要偏项目表达
- 二者面向的使用场景不同
- 一个回答“系统稳不稳”，一个回答“项目怎么讲”

### 问题 2：为什么当前只做这些基础指标？

回答要点：

- 先保证指标真实、稳定、可复用
- 当前项目还在持续迭代，复杂指标的数据基础还不够完整
- 先从通过率、命中率、路由匹配率开始最合理

### 问题 3：这个项目后面还能怎么继续增强指标表达？

回答要点：

- 加耗时统计
- 加工具成功率
- 加错误率
- 加 embedding 缓存命中率
- 加 Chroma 重建/复用比
