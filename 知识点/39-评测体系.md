# 第 39 课：评测体系

## 1. 本节目标

给主项目补上第一版真正可运行的评测体系，解决两个现实问题：

1. 你改完代码后，怎么知道主链路没退化
2. 这个项目后面怎么积累“可量化结果”

这一课的重点不是做学术论文式评测，而是做：

- 主项目最小回归评测
- 可重复运行
- 可保存结果

## 2. 是什么

这一步新增的是：

- 固定评测样例集
- 评测执行器
- 评测结果报告

对应文件：

- [evals/agent_eval_cases.json](/mnt/d/AIcodes/Agent/evals/agent_eval_cases.json)
- [src/evals.py](/mnt/d/AIcodes/Agent/src/evals.py)
- [src/run_evals.py](/mnt/d/AIcodes/Agent/src/run_evals.py)

它们分别负责：

- `agent_eval_cases.json`：保存样例
- `evals.py`：执行评测逻辑
- `run_evals.py`：命令行启动评测

## 3. 为什么

你现在这个项目已经不是最初的 Demo 了，已经有：

- `/qa`
- `/v1`
- Tool Calling
- ReAct
- 日志
- LangSmith

如果后面继续加功能，但没有评测，你会遇到很现实的问题：

- 路由变了但你没发现
- 问答命中率退化了但你没发现
- 以前能跑的样例改坏了

所以这一课的本质是：

**把“我觉得应该没问题”变成“我跑过固定样例，确认这次没退化”。**

## 4. 怎么做

### 第一步：先确定评测范围

当前第一版评测不贪大，先评最关键的两类能力：

- `qa`
- `v1`

为什么不一开始把所有能力都评掉？

因为当前最值钱的是：

- 知识点问答
- Agent v1 统一入口

这两块最容易在后续改动里被影响。

### 第二步：把样例独立成 JSON 文件

文件：

- [evals/agent_eval_cases.json](/mnt/d/AIcodes/Agent/evals/agent_eval_cases.json)

每条样例里会写：

- `case_id`
- `mode`
- `input_text`
- `expected_intent`
- `expected_result_type`
- `expected_notes`
- `expected_keywords`

这样后面你再扩样例时，不需要先改 Python 代码。

### 第三步：做最小评测执行器

文件：

- [src/evals.py](/mnt/d/AIcodes/Agent/src/evals.py)

这里先不追求花哨，而是实现：

- 跑样例
- 算通过/失败
- 统计通过率
- 保存结果

### 第四步：把结果输出成报告

输出文件：

- `data/evals/latest_eval_report.json`

这样每次评测后你都能看到：

- 总样例数
- 通过数
- 通过率
- 每条样例的细节

## 5. 关键代码

### 代码 1：评测样例对象

文件：

- [src/evals.py](/mnt/d/AIcodes/Agent/src/evals.py)

```python
@dataclass
class EvalCase:
    case_id: str
    mode: str
    input_text: str
    expected_intent: str | None = None
    expected_result_type: str | None = None
    expected_notes: list[str] | None = None
    expected_keywords: list[str] | None = None
```

这一步的意义是：

- 先把“什么叫一条评测样例”定义清楚

它不是一堆散乱字符串，而是一个结构明确的对象。

### 代码 2：关键词覆盖率

```python
def _keyword_coverage(self, text: str, keywords: list[str]) -> float:
    if not keywords:
        return 1.0
    hit_count = sum(1 for keyword in keywords if keyword in text)
    return hit_count / len(keywords)
```

这一版评测先不做复杂语义打分，而是先做一个最简单、最稳定的指标：

- 关键词覆盖率

这样做的好处是：

- 简单
- 结果稳定
- 适合当前阶段

缺点是：

- 语义能力有限

但这是合理的第一步。

### 代码 3：`qa` 样例评测

```python
note_hit = not expected_notes or any(note in selected_notes for note in expected_notes)
keyword_coverage = self._keyword_coverage(answer, case.expected_keywords or [])
passed = note_hit and keyword_coverage >= 0.5
```

当前对 `qa` 的通过标准是：

1. 是否命中了预期知识点文件
2. 答案是否覆盖了至少一半关键词

这不是完美评测，但已经能解决很关键的问题：

- 检索有没有明显跑偏
- 回答有没有严重退化

### 代码 4：`v1` 样例评测

```python
intent_match = case.expected_intent is None or result["intent"] == case.expected_intent
result_type_match = (
    case.expected_result_type is None
    or result["result_type"] == case.expected_result_type
)
```

这一步重点在评：

- 路由意图对不对
- 路由后的结果类型对不对

因为 `v1` 最核心的价值就是：

- 统一入口路由

所以第一版评测最该盯的就是它。

### 代码 5：统一生成报告

```python
report = {
    "generated_at": datetime.now().isoformat(timespec="seconds"),
    "total_cases": total,
    "passed_cases": passed,
    "pass_rate": round(passed / total, 4) if total else 0.0,
    "cases": case_results,
}
```

这里的目标很明确：

- 不只是打印几行 PASS/FAIL
- 而是沉淀成可保存、可回看、可后续统计的结构化结果

## 6. 常见错误

### 错误 1：只手工测，不做固定样例

这样每次都靠记忆，后面很容易遗漏退化问题。

### 错误 2：一开始就追求特别复杂的评测

比如：

- 多模型裁判
- 大量语义打分
- 复杂评估链

这对当前阶段来说太重了。

先做最小可用回归评测更合理。

### 错误 3：只测最终答案，不测路由

对 Agent 项目来说：

- 路由错了
- 工具没调对

也属于严重问题。

所以 `v1` 的意图匹配必须进入评测。

## 7. 常见面试问题

### 问题 1：为什么 Agent 项目需要评测体系？

回答要点：

- Agent 是多模块系统，改动容易产生回归
- 不能只靠手工试几次
- 评测体系能帮助确认能力是否退化

### 问题 2：为什么第一版评测不用更复杂的模型裁判？

回答要点：

- 当前目标是先做最小可用回归评测
- 关键词覆盖率和命中文件检查足够支撑第一版
- 后面可以逐步升级到更复杂的评估方式

### 问题 3：当前评测体系给项目带来了什么工程价值？

回答要点：

- 能稳定做回归验证
- 能积累通过率等量化指标
- 为后续简历里的“效果数据”提供基础
