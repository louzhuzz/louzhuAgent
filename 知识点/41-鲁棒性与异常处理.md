# 第 41 课：鲁棒性与异常处理

## 1. 本节目标

给主项目补上第一版真正有工程意味的鲁棒性处理。

这一课主要解决 4 类高频问题：

1. 模型接口偶发超时或网络抖动
2. 模型被短暂限流
3. 检索没有命中有效资料
4. 工具调用出错后直接把整个 Agent 链路打崩

这一课做完后，目标不是“永远不报错”，而是：

- 该重试的先重试
- 该降级的先降级
- 该返回友好错误的就别把栈追踪直接甩给用户

## 2. 是什么

这一步新增的不是单独一个新模块，而是对主链路的鲁棒性升级。

重点文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)
- [src/app_runtime.py](/mnt/d/AIcodes/Agent/src/app_runtime.py)
- [src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

对应的能力包括：

- 模型调用有限重试
- 模型调用统一超时
- 空检索兜底回答
- Tool Calling / ReAct 内部工具异常 observation 化
- 运行层统一异常包装

## 3. 为什么

一个 Agent 项目只要开始进入真实使用阶段，就一定会遇到这些问题：

- 网络偶发失败
- 平台瞬时限流
- 某次模型结构化输出不稳定
- 工具某一步参数错误
- 检索不到足够相关资料

如果不做鲁棒性处理，系统常见表现就是：

- 用户看见一长串异常
- 评测会被一条样例打断
- 多步 Agent 在中间某一步直接崩掉

所以这一课的本质不是“补点边角料”，而是：

**让主项目从“教学脚本”更接近“能扛住真实波动的工程系统”。**

## 4. 怎么做

### 第一步：把模型调用收口成统一重试入口

文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

当前项目所有能力最终都会走到：

- `_invoke_text(...)`
- `_invoke_json(...)`

所以最适合在这里做：

- 超时
- 重试
- API 异常包装

而不是每个业务模块自己各写一套。

### 第二步：把空检索当成正常分支处理

文件：

- [src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

空检索不是“程序异常”，而是：

**系统当前没找到足够相关的资料。**

所以它更适合返回友好提示，而不是直接抛异常。

### 第三步：把工具异常转成 Agent 可继续利用的 observation

文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

对于 Tool Calling / ReAct 来说，工具失败不一定意味着整条链路必须终止。

更合理的做法是：

- 把工具异常转成一条 observation
- 让模型看到这次工具失败了
- 再决定要不要换别的工具、换别的参数、或者直接结束

### 第四步：在运行层统一给用户友好错误

文件：

- [src/app_runtime.py](/mnt/d/AIcodes/Agent/src/app_runtime.py)

运行层本来就是：

- CLI
- Web

共用的统一入口，所以这里最适合把未预期异常统一包装成：

- 用户能看懂的错误消息
- 同时继续写入日志

## 5. 关键代码

### 代码 1：模型调用统一超时和有限重试

文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
self.model = ChatOpenAI(
    model=settings.model,
    api_key=settings.api_key,
    base_url=settings.base_url,
    temperature=0.7,
    request_timeout=30,
    max_retries=0,
)
```

这里的关键点是：

- `request_timeout=30`
  先给模型调用设一个明确超时上限
- `max_retries=0`
  不直接依赖底层 SDK 黑盒重试

因为后面我们要自己在 `_invoke_text(...)` 里做更清楚、更可控的重试逻辑。

### 代码 2：统一重试入口

```python
retryable_errors = (APIConnectionError, APITimeoutError, RateLimitError, APIError)
max_attempts = 3

for attempt in range(1, max_attempts + 1):
    try:
        response = self.model.bind(temperature=temperature).invoke(messages)
        return parse_text_output(response.content)
    except retryable_errors as exc:
        if attempt == max_attempts:
            raise RuntimeError(f"模型服务暂时不可用：{exc}") from exc
        time.sleep(0.8 * attempt)
```

这段代码的设计点有 3 个：

1. 只对“可重试”的异常重试  
   例如：
   - 连接失败
   - 超时
   - 限流
   - 一般 API 临时错误

2. 重试次数有限  
   不是无限重试，否则用户体验会更差。

3. 每次失败后短暂等待  
   用最简单的线性退避先顶住瞬时抖动。

### 代码 3：空检索兜底

文件：

- [src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

```python
if not retrieved_chunks:
    return {
        "selected_notes": [candidate.file_name for candidate in candidates],
        "retrieved_chunks": [],
        "index_statuses": index_statuses,
        "context": "",
        "answer": "当前候选知识点中没有检索到足够相关的资料，请尝试把问题问得更具体，或先用 /read 查看相关笔记。",
    }
```

这段代码的价值是：

- 不把“没检索到”当系统崩溃
- 直接返回对用户有用的下一步建议

这就是典型的：

**空结果兜底**

### 代码 4：工具错误 observation 化

文件：

- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
try:
    result = self.execute_tool(tool_name, arguments)
except Exception as exc:
    result = f"TOOL_ERROR: {exc}"
```

这段逻辑现在同时用在：

- `run_tool_calling_agent(...)`
- `run_react_agent(...)`

意思是：

- 工具失败时，不立刻把整个 Agent 终止
- 而是把失败结果保留下来
- 作为下一轮决策上下文的一部分

这是多步 Agent 很重要的一种鲁棒性设计。

### 代码 5：运行层统一包装未预期异常

文件：

- [src/app_runtime.py](/mnt/d/AIcodes/Agent/src/app_runtime.py)

```python
def _build_unexpected_error_result(prefix: str, exc: Exception) -> dict[str, Any]:
    return {
        "status": "error",
        "kind": "message",
        "message": f"{prefix}：{type(exc).__name__}: {exc}",
    }
```

然后在各个命令分支里补：

```python
except Exception as exc:
    error_result = _build_unexpected_error_result("知识点问答执行失败", exc)
    _log(logger, session_id, "knowledge_qa_error", error_result)
    return error_result
```

这样做的好处是：

- CLI 和 Web 都能拿到统一错误结构
- 日志继续有记录
- 用户不会看到原始 Python 崩溃栈直接灌出来

## 6. 常见错误

### 错误 1：所有异常都无脑重试

比如参数错误、文件不存在，这类问题重试没有意义。

应该只对：

- 网络抖动
- 超时
- 限流

这类瞬时错误重试。

### 错误 2：把空检索当程序异常

空检索很常见，应该返回：

- 资料不足
- 建议换个更具体的问题

而不是直接抛异常。

### 错误 3：工具一失败就整条 Agent 直接退出

这会让多步 Agent 很脆。

更合理的是：

- 记录失败 observation
- 让模型决定下一步

### 错误 4：只在 CLI 做异常处理

你当前项目有：

- CLI
- Web

所以异常包装应该尽量放在统一运行层，而不是只写在某一个入口里。

## 7. 常见面试问题

### 问题 1：为什么要把重试写在模型调用统一入口，而不是各业务模块自己写？

回答要点：

- 模型调用是横切关注点
- 放在统一入口更容易控制超时、重试次数和错误包装
- 避免每个业务模块重复写一套重试逻辑

### 问题 2：为什么工具失败后不直接终止 Agent？

回答要点：

- 多步 Agent 的目标是尽量完成任务，而不是一步失败就崩
- 工具失败本身也是观察结果
- 可以让模型基于失败 observation 改用其他工具或直接给出受限答案

### 问题 3：空检索为什么不抛异常？

回答要点：

- 空检索是业务结果，不一定是系统错误
- 用户更需要的是下一步建议
- 只有真正的程序故障才适合走异常分支
