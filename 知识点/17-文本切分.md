# 第十七步：文本切分

## 1. 本节目标

这一节要把“整篇文档”变成“适合检索的小块”。

本节新增的示例文件是：

- [examples/text_splitting_intro.py](/mnt/d/AIcodes/Agent/examples/text_splitting_intro.py)

完成后你会看到：

- 原始文档数
- 切分后的 chunk 数
- 每个 chunk 的内容预览和 metadata

## 2. 是什么

文本切分，就是把长文档拆成更小的文本块。

这些小块通常叫：

- chunk

在 RAG 里，真正进入后续向量化和检索阶段的，很多时候不是整篇文档，而是这些 chunk。

## 3. 为什么

如果你把整篇长文档直接拿去做检索，会出现几个问题：

- 单个文本太长，不利于精确召回
- 相关内容可能被无关内容稀释
- 后面拼接上下文时成本更高

所以 RAG 里通常不会直接检索“整篇文章”，而是检索：

**更小、更聚焦的文本块。**

这就是文本切分存在的意义。

## 4. 怎么做

### 第一步：先加载原始文档

这一节直接复用上一课的加载函数：

- [examples/document_loading_intro.py](/mnt/d/AIcodes/Agent/examples/document_loading_intro.py)

### 第二步：定义切分参数

最关键的两个参数是：

- `chunk_size`
- `chunk_overlap`

你可以先这样理解：

- `chunk_size`：每块最多多长
- `chunk_overlap`：相邻两块之间保留多少重叠内容

### 第三步：按窗口滑动切分

本节手写了一个最小切分器。

核心思路是：

1. 先取前 `chunk_size` 个字符
2. 下一块不要完全从头接着来，而是回退 `chunk_overlap`
3. 继续切下一块

这就是为什么：

```python
step = chunk_size - chunk_overlap
```

### 第四步：给每个 chunk 补 metadata

我们不只保留正文，还给每个 chunk 加了：

- `chunk_index`
- `chunk_start`
- `chunk_end`

这样后面你就能知道：

- 这是第几块
- 它来自原文的哪个位置

## 5. 关键代码

### 代码 1：切分参数校验

文件：[examples/text_splitting_intro.py](/mnt/d/AIcodes/Agent/examples/text_splitting_intro.py)

```python
if chunk_size <= 0:
    raise ValueError("chunk_size 必须大于 0。")
if chunk_overlap < 0:
    raise ValueError("chunk_overlap 不能小于 0。")
if chunk_overlap >= chunk_size:
    raise ValueError("chunk_overlap 必须小于 chunk_size。")
```

这段代码很重要，因为它防止了无意义参数。

比如：

- overlap 比 size 还大

那切分逻辑就会失效。

### 代码 2：滑动窗口切分

文件：[examples/text_splitting_intro.py](/mnt/d/AIcodes/Agent/examples/text_splitting_intro.py)

```python
step = chunk_size - chunk_overlap

while start < len(text):
    end = start + chunk_size
    chunk_text = text[start:end].strip()
```

这就是本节最核心的切分逻辑。

它的本质是：

- 每次取一个窗口
- 窗口之间保留部分重叠

### 代码 3：为 chunk 补 metadata

文件：[examples/text_splitting_intro.py](/mnt/d/AIcodes/Agent/examples/text_splitting_intro.py)

```python
chunk_metadata = {
    **document.metadata,
    "chunk_index": chunk_index,
    "chunk_start": start,
    "chunk_end": min(end, len(text)),
}
```

这一步的意义是：

- chunk 不是从零开始的孤立文本
- 它仍然保留和原文之间的关系

### 代码 4：切分后的 chunk 仍然是 Document

文件：[examples/text_splitting_intro.py](/mnt/d/AIcodes/Agent/examples/text_splitting_intro.py)

```python
chunks.append(Document(page_content=chunk_text, metadata=chunk_metadata))
```

这说明：

- 切分前是 `Document`
- 切分后仍然是 `Document`

只是现在从“整篇文档对象”变成了“文档块对象”。

## 6. 常见错误

### 错误 1：以为 chunk 越大越好

不一定。

chunk 太大时：

- 检索精度可能变差
- 上下文成本更高

### 错误 2：完全不设置 overlap

有时也能用，但容易把跨边界的重要信息切断。

所以 overlap 的作用是：

- 让相邻 chunk 之间有一点连续性

### 错误 3：在切分后丢失原文 metadata

后果：

- 后面检索到 chunk，却不知道它来自哪里

## 7. 常见面试问题

### 问题 1：为什么 RAG 要做文本切分？

回答要点：

- 长文档不适合直接高质量检索
- 切分后更容易精准召回相关片段
- 也更适合后续上下文拼接

深入追问：

- chunk 太大和太小分别会带来什么问题？

### 问题 2：`chunk_size` 和 `chunk_overlap` 分别控制什么？

回答要点：

- `chunk_size` 控制每块长度
- `chunk_overlap` 控制相邻块的重叠部分
- 二者共同影响召回质量和成本

深入追问：

- overlap 为什么不能大于等于 size？

### 问题 3：为什么 chunk 也要保留 metadata？

回答要点：

- 检索结果需要可追溯
- 需要知道 chunk 来自哪个文件、哪个位置
- 后面做引用和调试时非常重要

深入追问：

- 检索阶段最值得保留哪些 metadata？

### 问题 4：为什么这一节先手写切分器，而不是直接上框架切分器？

回答要点：

- 先理解切分本质
- 先理解 overlap 到底在做什么
- 后面再学框架切分器更容易理解参数

深入追问：

- 后面迁移到 `RecursiveCharacterTextSplitter` 的价值是什么？
