# 第三十三步：知识点问答系统（项目化重构版）

## 1. 本节目标

这一节要把前面零散出现过的：

- 读笔记
- 基于笔记回答
- 轻量筛选知识点文件

整理成主项目里的一个正式功能模块：

- 知识点问答系统

而且这次会明确遵守你现在项目里的一个现实约束：

- **要做真实 embedding**
- **但只对少量候选 `.md` 文件做 embedding，控制 token 消耗**

对应文件：

- [src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)
- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)
- [src/main.py](/mnt/d/AIcodes/Agent/src/main.py)
- [src/prompts.py](/mnt/d/AIcodes/Agent/src/prompts.py)

## 2. 是什么

这节做的是一个“主项目版 embedding 知识点问答系统”。

它的链路是：

1. 用户输入问题
2. 系统从全部知识点文件里挑少量候选 `.md`
3. 只对这些候选文件切块并做 embedding
4. 在候选范围内做向量检索
5. 把命中的资料片段拼进 Prompt
6. 模型基于这些资料回答

你要注意，这一版不是“全库 embedding”，而是：

- **少量候选文件 + 真实 embedding 检索**

## 3. 为什么

如果现在直接把项目主问答全都切成“全量 embedding + 全量检索”，会有两个问题：

1. 你的 embedding token 消耗会很快
2. 主项目复杂度会上升太快

所以当前最合理的项目化路线是：

- 主项目里用真实 embedding
- 但只在少量候选知识点范围内做检索
- 用这种方式兼顾效果和成本

这也是为什么这节会采用：

- 少量文件筛选
- 少量候选文件 embedding
- 候选范围内向量检索
- 基于证据回答

## 4. 怎么做

### 第一步：新增请求对象

我们新增了：

- `KnowledgeQARequest`

它目前包含：

- `question`
- `max_notes`

也就是说，这个功能不再只是“传一个字符串就完事”，而是开始具备项目化输入边界。

### 第二步：新增服务层

我们新增了：

- `KnowledgeQAService`

它负责：

1. 校验请求
2. 挑选候选知识点文件
3. 只对候选文件切块并构造临时向量库
4. 在候选范围内检索最相关片段
5. 构造上下文
6. 调模型回答

### 第三步：先做轻量候选筛选，再做 embedding

当前这一版不是直接对全部知识点做 embedding。

而是先通过：

- 英文词
- 中文短语
- 中文二元片段

去和文件名做轻量匹配，然后只取少量候选文件。

再只对这些候选文件做 embedding 检索。

这一步的核心目标不是“最强召回”，而是：

- 成本可控
- 容易解释
- 比单纯文件名匹配更接近真正 RAG

### 第四步：只对少量候选文件建临时向量库

当前默认只会选：

- `max_notes=3`

并且服务层里限制：

- 最多只允许读取 5 篇知识点

这是为了控制：

- embedding token 消耗
- Prompt 上下文长度
- 主项目复杂度

### 第五步：把问答接进主项目命令

现在你可以直接使用：

```text
/qa 什么是输出解析器？
```

这和旧的 `/study` 有区别：

- `/study` 更像“早期工具型示例”
- `/qa` 是“主项目版知识点问答功能”

### 代码 2：请求校验

文件：[src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

```python
def _validate_request(self, request: KnowledgeQARequest) -> KnowledgeQARequest:
    request.question = request.question.strip()
    if not request.question:
        raise ValueError("问题不能为空。")
    if request.max_notes <= 0:
        raise ValueError("max_notes 必须大于 0。")
    if request.max_notes > 5:
        raise ValueError("为了控制上下文和 token，当前最多只读取 5 篇知识点。")
```

这一段直接体现了你当前项目的现实约束：

- 不要让知识点问答无节制读取太多文件

### 代码 3：轻量文件打分

文件：[src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

```python
def _score_note_name(self, question: str, file_name: str) -> float:
    query_terms = self._extract_query_terms(question)
    if not query_terms:
        return 0.0

    lowered_name = file_name.lower()
    hit_count = sum(1 for term in query_terms if term in lowered_name)
    return hit_count / len(query_terms)
```

这一步的本质是：

- 先用问题关键词和文件名做轻量匹配

这样做不是为了代替 embedding，而是为了：

- **先缩小 embedding 范围**

### 代码 5：少量候选文件切块

```python
def build_candidate_documents(self, candidates: list[NoteCandidate]) -> list[Document]:
    documents: list[Document] = []
    for candidate in candidates:
        note_content = self.read_note(candidate.file_name)
        documents.extend(self._split_note_to_documents(candidate.file_name, note_content))
```

这段代码的重点是：

- 先筛文件
- 再切块
- 但始终只在少量候选文件范围内工作

也就是说，当前主项目的知识点问答默认不会全量对所有笔记做 embedding。

### 代码 6：把命中资料拼成上下文

文件：[src/knowledge_qa.py](/mnt/d/AIcodes/Agent/src/knowledge_qa.py)

```python
def build_context(self, candidates: list[NoteCandidate]) -> str:
    parts: list[str] = []
    for index, candidate in enumerate(candidates, start=1):
        note_content = self.read_note(candidate.file_name)
        parts.append(
            f"[资料 {index}] 文件: {candidate.file_name} | score={candidate.score:.4f}\n"
            f"{note_content}"
        )
```

你要注意这里有两个项目化设计点：

1. 会标出资料编号
2. 会标出文件名

这能帮助后面继续升级成：

- 引用回答
- 可追溯问答

## 6. 常见错误

### 错误 1：要么全量 embedding，要么完全不用 embedding

不是。

当前这节走的是一个中间方案：

- **少量候选文件 + embedding 检索**

### 错误 2：为了“更准”就默认读取所有 `.md`

这会直接把：

- token 消耗
- 上下文长度
- 响应成本

都拉高。

### 错误 3：已经做了候选筛选，又把命中文档整篇塞给模型

这样会浪费 embedding 检索的价值。

当前正确做法是：

- 只把命中的 chunk 送给模型

### 错误 4：只输出最终回答，不暴露选中的文件

这样你很难判断：

- 系统到底基于什么资料在回答

## 7. 常见面试问题

### 问题 1：为什么当前主项目知识问答不做全量 embedding，而是只做候选范围内 embedding？

回答要点：

- 因为 embedding 成本真实存在
- 先缩小候选范围再做 embedding，能兼顾效果和成本
- 这比纯文件名匹配更强，也比全量 embedding 更省

### 问题 2：为什么要把知识点问答也抽成服务层？

回答要点：

- 避免 `LearningAgent` 职责过重
- 方便后面升级检索策略
- 更符合项目化分层

### 问题 3：这一版和完整 RAG 的关系是什么？

回答要点：

- 这一版已经具备 RAG 的核心检索链
- 但它只在少量候选文件范围内运行
- 后面如果要升级，全量检索逻辑最自然还是替换 `KnowledgeQAService` 内部实现
