# 第三步：最小项目骨架

## 1. 本节目标

这一节的目标不是做复杂功能，而是：

**先把一个最小可运行的 Agent 主链路搭起来。**

你要做到的结果是：

- 能读取配置
- 能读取系统 Prompt
- 能接收用户输入
- 能调用模型返回答案

## 2. 是什么

“最小项目骨架”就是：

在功能尽量少的前提下，把项目运行必需的结构先搭好。

在我们这个项目里，最小骨架包括：

- `src/main.py`：程序入口
- `src/agent.py`：封装模型调用
- `src/config.py`：读取配置
- `src/prompts.py`：读取 Prompt 文件
- `prompts/system_prompt.txt`：系统提示词
- `.env.example`：配置示例
- `requirements.txt`：依赖列表

## 3. 为什么

因为 Agent 项目最怕一开始把所有能力混在一起。

如果你一开始就同时做：

- Tool Calling
- RAG
- 记忆
- 多 Agent

你会很难判断错误来自哪里。

所以第一原则是：

**先让主链路跑通，再逐层加能力。**

## 4. 怎么做

### 第一步：拆出配置层

把这些内容单独管理：

- API Key
- Base URL
- Model

这样主程序不会和平台配置耦合在一起。

### 第二步：拆出 Prompt 层

不要把系统提示词硬编码在 `main.py` 里。

把它放进独立文件：

- 方便改
- 方便版本管理
- 方便后续做 Prompt 实验

### 第三步：封装 Agent 类

Agent 类负责一件明确的事：

**接收用户问题，调用模型，返回回答。**

这一步不要急着把别的责任塞进去。

### 第四步：入口文件只负责交互

`main.py` 只做：

1. 启动程序
2. 读取输入
3. 调用 Agent
4. 打印结果

这样职责就清楚了。

## 5. 当前目录结构怎么理解

```text
Agent/
```

每一层职责分别是：

- `src/`：放程序逻辑
- `prompts/`：放提示词
- `知识点/`：放教学文档

## 6. 关键代码

### 代码 1：读取配置

文件：[src/config.py](/mnt/d/AIcodes/Agent/src/config.py)

```python
@dataclass
class Settings:
    api_key: str
    base_url: str
    model: str


def load_settings() -> Settings:
    api_key = os.getenv("ARK_API_KEY", "").strip()
    base_url = os.getenv("ARK_BASE_URL", "https://ark.cn-beijing.volces.com/api/v3").strip()
    model = os.getenv("ARK_MODEL", "").strip()
```

这段代码的作用：

- 用 `Settings` 统一管理配置
- 从环境变量读取火山方舟配置
- 给 `base_url` 提供默认值

你要理解的点：

- 配置不是业务逻辑
- 配置和代码分离是工程习惯

### 代码 2：读取系统 Prompt

文件：[src/prompts.py](/mnt/d/AIcodes/Agent/src/prompts.py)

```python
PROMPTS_DIR = Path(__file__).resolve().parent.parent / "prompts"


def load_system_prompt() -> str:
    prompt_path = PROMPTS_DIR / "system_prompt.txt"
    return prompt_path.read_text(encoding="utf-8").strip()
```

这段代码的作用：

- 找到 `prompts/` 目录
- 读取系统提示词
- 返回给 Agent 使用

你要理解的点：

- Prompt 是可以独立维护的资源
- Prompt 和 Python 逻辑应该解耦

### 代码 3：封装最小 Agent

文件：[src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)

```python
class LearningAgent:
    def __init__(self, settings: Settings):
        self.settings = settings
        self.client = OpenAI(
            api_key=settings.api_key,
            base_url=settings.base_url,
        )
        self.system_prompt = load_system_prompt()
```

这段代码的作用：

- 初始化模型客户端
- 保存配置
- 读取系统 Prompt

这就是最小 Agent 的初始化过程。

### 代码 4：程序入口

文件：[src/main.py](/mnt/d/AIcodes/Agent/src/main.py)

```python
def main() -> None:
    settings = load_settings()
    agent = LearningAgent(settings)

    print("个人学习助理 Agent 已启动。输入 exit 退出，输入 /clear 清空对话记忆。")
```

这段代码的作用：

- 先读配置
- 再创建 Agent
- 然后进入交互循环

## 7. 常见错误

### 错误 1：把所有代码都堆在 `main.py`

后果：

- 代码越来越乱
- 后面很难扩功能

### 错误 2：把 Prompt 写死在代码里

后果：

- 改 Prompt 很不方便
- 无法做 Prompt 版本管理

### 错误 3：把配置写死在源码里

后果：

- 不安全
- 切换环境麻烦
- 容易误提交密钥

## 8. 常见面试问题

### 问题 1：为什么要先搭最小项目骨架，而不是直接上框架？

回答要点：

- 先理解底层调用链
- 先分清配置、Prompt、业务逻辑的职责
- 先把最小闭环跑通，后面再上框架才知道框架解决了什么

深入追问：

- 如果现在就上 LangChain，会丢失哪些底层理解？

### 问题 2：Prompt 为什么要单独放文件？

回答要点：

- Prompt 变更频率高
- 它本质上是“配置型资源”
- 单独管理更利于维护、测试和版本对比

深入追问：

- Prompt 应该版本化吗？
- Prompt 和代码发布是否要同步？

### 问题 3：配置层为什么要单独抽出来？

回答要点：

- 配置和业务逻辑分离
- 便于多环境切换
- 便于统一校验和报错

深入追问：

- 如果以后要支持多个模型平台，配置层应该怎么扩展？
