# 第三十四步：任务拆解器（项目化重构版）

## 1. 本节目标

这一节要把“复杂目标拆步骤”做成主项目里的正式能力模块。

也就是说，它不再只是：

- 聊天时让模型顺手列几个步骤

而是会升级成：

- 有请求对象
- 有服务层
- 有结构化 JSON 输出
- 有基本结果校验

对应文件：

- [src/task_breakdown.py](/mnt/d/AIcodes/Agent/src/task_breakdown.py)
- [src/agent.py](/mnt/d/AIcodes/Agent/src/agent.py)
- [src/main.py](/mnt/d/AIcodes/Agent/src/main.py)
- [src/prompts.py](/mnt/d/AIcodes/Agent/src/prompts.py)
- [prompts/task_breakdown_prompt.txt](/mnt/d/AIcodes/Agent/prompts/task_breakdown_prompt.txt)

## 2. 是什么

任务拆解器的目标是：

**把一个较大的学习目标，拆成一组可执行、可验证的步骤。**

它和学习计划生成器相似，但关注点不同：

- 学习计划生成器更偏“按天安排”
- 任务拆解器更偏“按步骤拆目标”

比如：

- `/plan LangChain | 零基础 | 5 | 做出一个可运行 Demo`
  更像安排 5 天学习计划

- `/breakdown 做一个个人学习助理 | 零基础 | 14 | 可执行步骤`
  更像把“大目标”拆成实现路径

## 3. 为什么

主项目后面会继续整合：

- 学习计划
- 知识点问答
- Agent 能力

如果没有“任务拆解器”，用户面对大目标时只能得到：

- 一个笼统建议
- 或一个泛泛的学习计划

但很多真实任务更需要的是：

- 第一步做什么
- 第二步做什么
- 每一步产出是什么

所以任务拆解器是主项目里一个非常自然的项目化能力。

## 4. 怎么做

### 第一步：定义请求对象

我们新增了：

- `TaskBreakdownRequest`

它包含：

- `goal`
- `current_level`
- `available_days`
- `output_style`

这让“任务拆解”从一段散乱文本，变成了结构化输入。

### 第二步：新增服务层

我们新增了：

- `TaskBreakdownService`

它负责：

1. 校验请求
2. 渲染 Prompt
3. 调模型拿结构化 JSON
4. 校验结果

### 第三步：设计专用 Prompt

我们新增了：

- [prompts/task_breakdown_prompt.txt](/mnt/d/AIcodes/Agent/prompts/task_breakdown_prompt.txt)

这个 Prompt 会要求模型返回：

- `summary`
- `steps`
- `risks`

其中 `steps` 里的每一项都包含：

- `title`
- `goal`
- `actions`
- `expected_output`

### 第四步：新增命令入口

现在你可以直接用：

```text
/breakdown 做一个个人学习助理 | 零基础 | 14 | 可执行步骤
```

这会返回结构化 JSON。

## 5. 关键代码

### 代码 1：任务拆解请求对象

文件：[src/task_breakdown.py](/mnt/d/AIcodes/Agent/src/task_breakdown.py)

```python
@dataclass
class TaskBreakdownRequest:
    goal: str
    current_level: str = "零基础"
    available_days: int = 7
    output_style: str = "可执行步骤"
```

这段代码的核心作用是：

- 把任务拆解的输入参数明确化

这样后面如果你要增加：

- 风险偏好
- 结果粒度
- 是否需要里程碑

都可以自然扩展。

### 代码 2：服务层

文件：[src/task_breakdown.py](/mnt/d/AIcodes/Agent/src/task_breakdown.py)

```python
class TaskBreakdownService:
    def __init__(
        self,
        invoke_json: Callable[[str, float], dict[str, Any]],
        render_prompt: Callable[[TaskBreakdownRequest], str],
    ) -> None:
        self.invoke_json = invoke_json
        self.render_prompt = render_prompt
```

这里和学习计划生成器保持同样的项目化风格：

- 模型调用和 Prompt 渲染通过依赖注入传入
- 服务层不自己创建模型

### 代码 3：结果校验

文件：[src/task_breakdown.py](/mnt/d/AIcodes/Agent/src/task_breakdown.py)

```python
def _validate_result(
    self,
    result: dict[str, Any],
    request: TaskBreakdownRequest,
) -> dict[str, Any]:
    steps = result.get("steps")
    if not isinstance(steps, list) or not steps:
        raise ValueError("任务拆解结果必须包含非空的 steps 列表。")

    if len(steps) > request.available_days:
        raise ValueError(
            f"模型返回了 {len(steps)} 个步骤，但可用天数只有 {request.available_days}。"
        )
```

这一段体现了主项目最重要的原则之一：

**模型输出必须经过项目层校验。**

## 6. 常见错误

### 错误 1：把任务拆解器和学习计划生成器当成同一个东西

不是。

两者有交叉，但关注点不同：

- 学习计划：时间安排
- 任务拆解：目标分解

### 错误 2：觉得拆得越细越好

不一定。

如果拆得太细，用户反而难以执行。

### 错误 3：让模型无限拆步骤

这会让输出变得冗长，而且不受控。

所以当前项目版会限制：

- 步骤数不能超过可用天数

### 错误 4：不做结构校验

主项目功能不能把模型输出当成绝对可靠。

## 7. 常见面试问题

### 问题 1：任务拆解器和学习计划生成器的区别是什么？

回答要点：

- 学习计划偏时间维度
- 任务拆解偏目标分解维度
- 两者可以组合，但不应混为一体

### 问题 2：为什么任务拆解器也要走服务层？

回答要点：

- 保持主项目能力模块风格一致
- 降低 `LearningAgent` 的复杂度
- 便于后面继续扩展与测试

### 问题 3：为什么要限制步骤数量不超过可用天数？

回答要点：

- 让拆解结果和现实资源约束一致
- 避免模型输出无节制膨胀
- 增强结果可执行性

如果这三点成立，这一节就完成了。
