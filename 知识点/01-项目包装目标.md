# 项目包装目标

这份文件用于记录当前项目后续迭代时要持续对齐的**简历项目目标模板**。

它不是替代：

- [知识点/00-学习路线指导.md](/mnt/d/AIcodes/Agent/知识点/00-学习路线指导.md)
- [知识点/07-教学总纲.md](/mnt/d/AIcodes/Agent/知识点/07-教学总纲.md)

而是专门回答一个问题：

**这个项目后面要朝什么样的“可投递、可面试、可量化表达”的方向持续完善。**

---

## 1. 当前对齐目标

后续这个项目要尽量往下面这类项目描述靠近：

### 项目名称

个人学习助理 Agent 开发

### 项目周期

按你真实简历时间填写。

### 技术栈

- `LLM`：火山方舟 / 豆包 / 兼容 OpenAI 接口模型
- `Agent`：ReAct、Tool Calling、统一路由、后续可补 Plan-Execute
- `RAG`：Embedding、Retriever、Rerank、ChromaDB
- `框架`：LangChain 1.x、后续可补 LangGraph
- `工程`：Python、API 适配层、Memory、日志、评测、缓存

---

## 2. 模板目标拆解

后面每次做功能，不只是问“能不能跑”，还要问：

- 能不能支撑简历里的技术栈表达
- 能不能支撑面试里的项目讲述
- 能不能支撑指标化结果

### 2.1 工具定义与适配

目标：

- 不只是有 1 到 2 个教学型工具
- 而是形成一组主项目可讲的核心工具

后续应逐步补齐：

- 知识点文件检索
- 知识点文件读取
- 多篇笔记总结
- 学习计划生成
- 任务拆解
- 后续可继续扩展搜索、规则校验、输出检查类工具

技术要求：

- 每个工具都要有明确输入输出
- 统一 `schema`
- 有统一工具执行入口
- 有工具适配层

简历关键词：

- `tool schema`
- `adapter`
- `tool registry`
- `tool routing`

### 2.2 控制策略与工作流设计

目标：

- 不只是单轮工具调用
- 而是能讲清楚完整决策链

后续应逐步补齐：

- `Tool Router`
- `ReAct`
- `Plan-Execute`
- 多步工具调用闭环
- 条件分支和失败回退

简历关键词：

- `Tool Router`
- `ReAct`
- `Plan-Execute`
- `workflow`
- `orchestration`

### 2.3 决策逻辑

目标：

- 形成“规则 + LLM”混合决策机制

后续应逐步补齐：

- 关键词触发
- 高优先级规则兜底
- LLM 自主判断是否需要工具
- 多轮调用链
- 上下文状态更新

简历关键词：

- `intent classification`
- `rule + llm hybrid routing`
- `multi-step tool calling`

### 2.4 Memory 与上下文

目标：

- 当前已有短期记忆
- 后面要补到可面试表达的 Memory 模块

后续应逐步补齐：

- 短期会话记忆
- 长期记忆
- 工具调用过程记录
- 可复用上下文状态

简历关键词：

- `short-term memory`
- `long-term memory`
- `state management`

### 2.5 监控、日志与评测

目标：

- 这是当前项目最欠缺、但最能拉开面试差距的一块

后续必须补齐：

- 工具调用日志
- 请求耗时
- 成功率
- 失败原因
- Prompt / Tool / Response trace
- 评测集
- 回归测试

简历关键词：

- `observability`
- `tracing`
- `eval`
- `fallback`
- `retry`

### 2.6 成本优化

目标：

- 当前项目已经有真实约束：embedding token 昂贵

后续必须持续强化：

- 少量候选文件筛选
- embedding 缓存
- ChromaDB 持久化
- 增量更新
- 控制 `top_k`
- 控制 chunk 数

简历关键词：

- `cost optimization`
- `embedding cache`
- `candidate filtering`
- `persistent vector store`

## 7. 结合 2026 校招趋势的补充要求

这一部分是结合你刚发的那篇“27届实习校招必看：后端&大模型应用开发突围指南”提炼出来的。

重点不是照搬文章，而是把对当前项目真正有用的内容固化成后续要求。

### 7.1 项目定位要更贴近“后端 + 大模型应用开发”

后面包装这个项目时，不要只把它说成：

- 一个 Agent 学习项目

而要逐步往下面这个定位靠：

- 一个具备后端工程思维的大模型应用项目

也就是说，后面要持续补这几类能力：

- 请求入口与服务层
- 工具适配与统一调度
- RAG 检索链
- 成本控制
- 日志、观测、评测

这会让项目更符合现在岗位里的关键词：

- `后端 + AI`
- `大模型应用开发`
- `Agent 工程化`

### 7.2 后续技术栈表达要更完整

文章里反复强调一个现实：

- 面试官不会只看你“会不会调 API”
- 而会看你是不是具备完整应用开发视角

所以后面这个项目在简历里，技术栈要逐步补成更完整的表达：

- `Python`
- `LLM API`
- `LangChain 1.x`
- `ReAct / Tool Calling / Agent Routing`
- `RAG`
- `Embedding`
- `ChromaDB`
- `Memory`
- `Tracing / Eval`

如果后面再继续扩展，可以再补：

- `FastAPI`
- `Docker`
- `Redis`

但当前项目主线仍然先放在：

- `Agent`
- `RAG`
- `工程化`

### 7.3 RAG 这一块必须朝“可面试表达”继续升级

文章里强调：

- 文档切片
- Embedding 选型
- Rerank
- 幻觉抑制
- 引用溯源
- 成本优化

这些恰好也是你当前项目后面最应该补的。

所以后面 RAG 相关能力不能只停留在：

- 能检索
- 能回答

而要进一步补成：

- Embedding 缓存
- ChromaDB 持久化
- 引用溯源
- 日志和效果评估
- 成本控制说明

### 7.4 Agent 部分不能只讲“会用”，还要讲“怎么控”

文章里提到的 Agent 面试高频问题，本质都在问同一件事：

**你怎么让 Agent 不失控。**

所以后面这个项目还要持续补这些表达：

- 怎么防止死循环
- 怎么限制最大步数
- 工具参数错误怎么办
- 工具失败怎么办
- 什么时候降级到普通回答
- 什么时候必须走工具

也就是说，后面你做日志系统、评测体系时，要始终围绕：

- `稳定性`
- `可控性`
- `可回溯`

这三个关键词。

### 7.5 量化结果不一定非要业务大盘，也可以先做工程指标

文章里强调“尽量写量化结果”，这一点是对的。

但你现在是个人项目，未必一开始就有：

- 日请求量
- 用户满意度
- 真正线上业务转化

所以你后面优先要积累的是“工程型指标”，例如：

- 检索命中率提升
- 首次回答耗时
- 工具调用成功率
- 平均工具步数
- 缓存命中率
- embedding 调用次数下降比例
- token 成本下降比例

这类指标同样可以成为简历里的亮点。

### 7.6 这个项目后面要逐步靠近的最终表达

结合文章趋势，你后面要把这个项目逐步打磨成这种感觉：

- 不是“我做了一个会聊天的 AI”
- 而是“我做了一个具备 RAG、Tool Calling、统一路由、成本优化、可观测能力的个人学习助理 Agent”

这两种表达的差距非常大。

前者更像练手 Demo，后者更像可面试、可包装、可继续扩展的工程项目。
